# Deployment Guide

## Overview

The Jupyter Notebook Validator Operator is a Kubernetes-native operator that automates
Jupyter Notebook validation in MLOps workflows. Built with Go 1.22, Operator SDK v1.37.0,
and controller-runtime, it orchestrates notebook execution in isolated pods, performs golden
notebook comparisons, integrates with Git repositories, and validates against deployed ML
models (KServe, OpenShift AI, vLLM, TorchServe, TensorFlow Serving, Triton, Ray Serve,
Seldon, BentoML).

The operator targets OpenShift 4.18+ and Kubernetes 1.31+. It supports four deployment
methods (Kustomize, Helm, OLM, raw manifests) and publishes to OperatorHub.io and
OpenShift/OKD OperatorHub via community-operators repositories.

Repository: https://github.com/tosin2013/jupyter-notebook-validator-operator
Provider: Decision Crafters (https://www.decisioncrafters.com/)
Maintainer: Tosin Akinosho <takinosh@redhat.com>

## Prerequisites

Required tools for building and deploying the operator:

- Go 1.22+ (toolchain 1.24.10 in go.mod)
- Docker or Podman (set CONTAINER_TOOL=podman if using Podman)
- kubectl or oc CLI
- make
- kustomize v5.3.0 (auto-downloaded by Makefile)
- controller-gen v0.14.0 (auto-downloaded by Makefile)

For Helm deployment:
- Helm 3.13+
- cert-manager (required for webhooks)

For OLM bundle/catalog workflows:
- operator-sdk v1.37.0 (auto-downloaded by Makefile)
- opm v1.43.0 (auto-downloaded by Makefile)

For local testing:
- Kind v0.23.0+ (for local Kubernetes clusters)
- Ginkgo v2.19.0 (Go test framework, in go.mod)
- setup-envtest (auto-downloaded by Makefile, uses K8s 1.29.0 assets)

For CI/CD:
- GitHub Actions (workflows in .github/workflows/)
- Quay.io account (for image registry)
- Codecov account (for coverage reports)

## Deployment Steps

### Method 1: Kustomize (recommended for development)

1. Install CRDs into the cluster:
   make install

2. Build and push the operator image:
   make docker-build docker-push IMG=quay.io/takinosh/jupyter-notebook-validator-operator:latest

3. Deploy the operator:
   make deploy IMG=quay.io/takinosh/jupyter-notebook-validator-operator:latest

4. Verify the deployment:
   kubectl get pods -n jupyter-notebook-validator-operator-system

5. Apply a sample validation job:
   kubectl apply -f config/samples/mlops_v1alpha1_notebookvalidationjob.yaml

To undeploy:
   make undeploy

### Method 2: Helm Chart

1. Sync CRDs to the Helm chart:
   make helm-sync-crds

2. Install the Helm chart:
   helm install jupyter-validator helm/jupyter-notebook-validator-operator \
     --namespace jupyter-validator-system \
     --create-namespace \
     --set openshift.enabled=true \
     --set prometheus.enabled=true \
     --wait --timeout 10m

   Or use the Makefile shortcut:
   make helm-install

3. Verify:
   kubectl get pods -n jupyter-validator-system

To upgrade:
   make helm-upgrade

To uninstall:
   make helm-uninstall

### Method 3: OLM Bundle (for OperatorHub)

1. Generate the OLM bundle:
   make bundle IMG=quay.io/takinosh/jupyter-notebook-validator-operator:1.0.6

2. Build and push the bundle image:
   make bundle-build bundle-push BUNDLE_IMG=quay.io/takinosh/jupyter-notebook-validator-operator-bundle:1.0.6

3. Test the bundle locally with operator-sdk:
   operator-sdk run bundle quay.io/takinosh/jupyter-notebook-validator-operator-bundle:1.0.6

4. Build and push the FBC catalog image (optional):
   make catalog-build catalog-push CATALOG_IMG=quay.io/takinosh/jupyter-notebook-validator-operator-catalog:v1.0.6

### Method 4: Raw Manifests

1. Generate consolidated install YAML:
   make build-installer IMG=quay.io/takinosh/jupyter-notebook-validator-operator:latest

2. Apply the generated manifest:
   kubectl apply -f dist/install.yaml

### Running Locally (development)

Run the operator on your host machine against a cluster:
   make run

This requires a valid kubeconfig and CRDs installed via `make install`.

## OperatorHub Catalog Submission and PR Approval

The operator is published to two community-operators repositories to appear in
OperatorHub.io (Kubernetes) and OpenShift/OKD OperatorHub.

### Target Repositories

1. k8s-operatorhub/community-operators (OperatorHub.io / Kubernetes catalog)
   - Fork: https://github.com/tosin2013/community-operators
   - Multiple bundle versions may be submitted in a single PR
   - Merged PR #7235 added versions 1.0.7, 1.0.8, 1.0.9
   - Makes operator visible on https://operatorhub.io

2. redhat-openshift-ecosystem/community-operators-prod (OpenShift/OKD catalog)
   - Fork: https://github.com/tosin2013/community-operators-prod
   - ONE bundle per PR (strict requirement; submitting multiple versions fails CI)
   - Makes operator visible in the OpenShift/OKD OperatorHub built-in console
   - Currently only v1.0.2 exists here; newer versions needed for OKD visibility

### Bundle Submission Workflow

Step-by-step process for submitting to either community-operators repository:

1. Generate and validate the OLM bundle:
   make bundle IMG=quay.io/takinosh/jupyter-notebook-validator-operator:{VERSION}
   operator-sdk bundle validate ./bundle
   operator-sdk bundle validate ./bundle --select-optional suite=operatorframework

2. Build and push the bundle image:
   make bundle-build bundle-push \
     BUNDLE_IMG=quay.io/takinosh/jupyter-notebook-validator-operator-bundle:{VERSION}

3. Run pre-submission tests locally:
   ./scripts/full-pre-submission-test.sh --version {VERSION}
   ./scripts/pre-submission-validate.sh ./bundle {VERSION}

4. Fork the target community-operators repository (if not already forked).

5. Clone the fork and create a branch:
   git clone https://github.com/tosin2013/community-operators.git
   cd community-operators
   git checkout -b add-jupyter-validator-{VERSION}

6. Copy bundle contents into the operator directory:
   mkdir -p operators/jupyter-notebook-validator-operator/{VERSION}
   cp -r /path/to/bundle/* operators/jupyter-notebook-validator-operator/{VERSION}/

7. Ensure ci.yaml exists in the operator root directory:
   cp bundle/ci.yaml operators/jupyter-notebook-validator-operator/ci.yaml
   The ci.yaml configures updateGraph: replaces-mode for the upgrade chain.

8. Commit with DCO sign-off (required):
   git config user.email "takinosh@redhat.com"
   git config user.name "Tosin Akinosho"
   git add operators/jupyter-notebook-validator-operator/{VERSION}/
   git commit -s -m "operator jupyter-notebook-validator-operator ({VERSION})"

9. Push and create the PR:
   git push origin add-jupyter-validator-{VERSION}
   PR title must be: operator jupyter-notebook-validator-operator ({VERSION})

### PR Approval Process

- PRs trigger automated CI pipelines in both community-operators repositories.
- CI validates bundle structure, runs operator-sdk bundle validate, and checks the
  OperatorHub validation suite.
- For community-operators-prod: CI is stricter and enforces one bundle per PR.
- DCO (Developer Certificate of Origin) sign-off is required on all commits.
  If missing, amend with: git rebase HEAD~1 --signoff && git push --force-with-lease
- Community maintainers review the PR after CI passes.
- The CSV must include a replaces field pointing to the previous version
  (e.g., replaces: jupyter-notebook-validator-operator.v1.0.5).
- The containerImage annotation must match the actual pushed operator image tag.

### Pre-Submission Checklist

Before submitting to community-operators repositories, verify:

- Provider name is "Decision Crafters" with URL https://www.decisioncrafters.com/
- Maintainer is "Tosin Akinosho" with email takinosh@redhat.com
- containerImage annotation matches the actual operator image
- replaces field in CSV points to previous version
- Commits are signed with DCO (git commit -s)
- Standard operator icon is used (Decision Crafters branded, not personal avatar)
- Bundle validates: operator-sdk bundle validate ./bundle
- OLM install tested: operator-sdk run bundle {BUNDLE_IMG}
- com.redhat.openshift.versions annotation is set (e.g., "v4.18-v4.22")

Verification commands:
   grep -A2 "provider:" bundle/manifests/*.clusterserviceversion.yaml
   grep -A2 "maintainers:" bundle/manifests/*.clusterserviceversion.yaml
   grep "containerImage:" bundle/manifests/*.clusterserviceversion.yaml
   grep "replaces:" bundle/manifests/*.clusterserviceversion.yaml

## Troubleshooting

### Git Clone Failures
Check that the credentials secret exists and has correct keys (username/password for
HTTPS or ssh-privatekey for SSH). Verify the secret is in the same namespace as the
NotebookValidationJob.

### Pod Stuck in Pending
Check resource quotas and node capacity. Ensure the specified container image exists
and is pullable. Check image pull secrets if using a private registry.

### Build Failures (S2I/Tekton)
Check requirements.txt syntax. Ensure the base image supports the required Python
dependencies. For S2I, check BuildConfig and Build logs with oc commands. For Tekton,
check PipelineRun status.

### Model Validation Failures
Verify the model exists and is ready in the target namespace. Check network policies
allow access from the validation pod to the model endpoint. Ensure the platform
(KServe, OpenShift AI, vLLM, etc.) is correctly detected.

### Operator Not Appearing in OKD OperatorHub
The operator only appears if submitted to redhat-openshift-ecosystem/community-operators-prod.
Currently only v1.0.2 exists there. Submit newer versions following the OperatorHub
Catalog Submission workflow above. See docs/CATALOG-VISIBILITY-INVESTIGATION.md.

### kube-rbac-proxy Image Not Found
gcr.io/kubebuilder/kube-rbac-proxy:v0.16.0 does not exist. Use
quay.io/brancz/kube-rbac-proxy:v0.18.0 instead. This fix has been applied to all
release branches (main, release-4.18, release-4.19, release-4.20).

### Bundle Submission Failures

DCO Check Failed:
  git rebase HEAD~1 --signoff
  git push --force-with-lease origin your-branch

containerImage Annotation Mismatch:
  Ensure the containerImage annotation in the CSV matches the actual pushed image tag.
  grep "containerImage:" bundle/manifests/*.clusterserviceversion.yaml

Multiple Bundles in community-operators-prod:
  community-operators-prod requires ONE bundle per PR. Create separate PRs for each version.

Channel Mismatch:
  Ensure bundle/metadata/annotations.yaml channel matches the channel in catalog/catalog.yaml
  (should both be "stable").

### Webhook Issues
If webhooks fail, ensure cert-manager is installed and the certificate is provisioned.
For OperatorHub.io deployments, webhooks may be disabled via ENABLE_WEBHOOKS=false.

### Watch Operator Logs
  kubectl logs -n jupyter-notebook-validator-operator-system \
    deployment/notebook-validator-controller-manager -f

### Inspect CR Status
  kubectl get notebookvalidationjob <name> -o yaml
  kubectl describe notebookvalidationjob <name>
